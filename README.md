# hands

A small project for generating SQL using LangChain + Groq models.  
It now exposes a simple FastAPI endpoint that wraps the existing
`query_tool/query_gen.py` logic so you can request SQL generation via HTTP.

## Usage

### As a script

```bash
python3 query_tool/query_gen.py
```

### As an API

1. ensure `GROQ_API_KEY` is exported or stored in `.env`
2. start the server:

```bash
uvicorn query_tool.api:app --reload
```

3. POST JSON to `/generate-sql` with the following schema:

```json
{
  "question": "...",
  "keywords": ["..."],
  "country": "Optional country",
  "languages": ["English", "Chinese"],
  "date": "2026-02-10"
}
```

The response will contain a single `sql` field with the generated query.

### Fetching data

To actually run the generated SQL against a database, a simple fetcher
utility is provided in `query_tool/fetcher.py`.  It connects to a configured
backend and returns rows as JSON-friendly dictionaries.

The API exposes a convenience endpoint:

```
POST /generate-and-fetch
```

It accepts the same body as `/generate-sql` but responds with both the
generated query and the query results.

Example response:

```json
{
  "sql": "SELECT ...",
  "results": [ {"id":1, "title":"..."}, ... ]
}
```

### Postgres support

The fetcher can now talk to PostgreSQL.  Simply export a `DATABASE_URL`
(example: `postgresql://user:pass@localhost:5432/mydb` or
`postgresql://user:pass@203.0.113.42:5432/mydb` for a host with a public IP)
before running the service and it will prefer Postgres over sqlite.  Make
sure `psycopg2-binary` is installed (it's added to the requirements file).

Using a public IP is fine as long as your database accepts connections from
 the service's network and the credentials are secure.  The URL can point to
 any reachable host, including cloud providers, on-premise servers, or
 managed instances.

If the variable isn't present the code still falls back to the local
`db.sqlite3` file.
### Required RPC for dynamic SQL

The fetcher executes whatever SQL is generated by the LLM via a Supabase
remote procedure call. To enable this you must define a helper function in
your database. In the SQL editor run something like:

```sql
create or replace function public.execute_sql(query text)
returns setof jsonb as $$
begin
  return query execute query;
end;
$$ language plpgsql;
```

This function simply takes a text query and returns the results as JSON.  You
can customize it (e.g. aggregate into a single JSON blob) but the name and
signature must match what the client calls:

```python
client.rpc("execute_sql", {"query": sql}).execute()
```

Once the function exists, `/generate-and-fetch` will run the dynamically
created SQL against your Supabase database.  No additional modifications to
the Python code are necessary.
